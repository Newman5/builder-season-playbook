# Piece of Pie GitHub Activity Dashboard — MVP Spec

## Purpose

The Piece of Pie GitHub Activity Dashboard is a lightweight static website that helps hackathon organizers track public GitHub activity from registered builders.

The goal is not to judge builders by commit count alone. The goal is to create a simple, visible, open-web-friendly feed of building activity that supports hackathon operations, weekly check-ins, storytelling, and final review.

This dashboard should answer questions like:

- Who has been active on GitHub this week?
- What projects are showing recent build momentum?
- Which repositories are associated with each builder?
- When was each builder's most recent public GitHub activity?
- Where can organizers quickly find each builder's project, GitHub profile, and weekly evidence?

This MVP should stay intentionally simple.

---

## Core Idea

The site has two main data files:

```txt
config/repos.yml  # manually maintained builder registry
builders.json     # normalized site data generated from the registry
activity.json     # generated snapshot of recent GitHub activity
```

The static site reads these files and displays:

1. A dashboard/home page with all builders
2. A recent activity feed
3. Individual builder pages
4. Basic weekly activity indicators

---

## MVP Principle

Start with the least complicated system that is useful.

This means:

- No database
- No login system
- No private data
- No GitHub webhooks at first
- No real-time updates
- No complex scoring
- No judging based only on commits

The MVP should work from a simple list of public GitHub accounts and public repositories.

---

## Suggested Tech Stack

Recommended stack:

- Static site generator: 11ty or Astro
- Data files: JSON
- Automation: GitHub Actions
- API source: GitHub public API
- Hosting: GitHub Pages

This keeps the system open, inspectable, forkable, and easy to adapt for future hackathons.

## Data Model

### config/repos.yml

This YAML file is manually maintained by the hackathon organizers.

Example:

```yml
repos:
  - id: alice
    name: Alice Example
    github: alice-dev
    x: alicebuilds
    project_name: Open Wallet Tool
    project_url: https://example.com
    repo_url: https://github.com/alice-dev/open-wallet-tool
    pies: [builders, feedback]
    notes: Registered builder. Tracking one main repo.
```

### Field Notes

| Field         | Required? | Description                                                   |
| ------------- | --------- | ------------------------------------------------------------- |
| `id`          | Yes       | Internal slug used for builder page URLs                      |
| `name`        | Yes       | Display name                                                  |
| `github`      | Yes       | GitHub username                                               |
| `x`           | No        | X/Twitter handle, without @                                   |
| `projectName` | No        | Name of the builder's project                                 |
| `projectUrl`  | No        | Website, demo, or project page                                |
| `repo_url`    | Yes       | Single tracked repository for Phase 1 commit counting         |
| `pies`        | No        | Pies the builder may be participating in                      |
| `notes`       | No        | Admin notes or public display notes, depending on use         |

Important rule: Phase 1 tracks one repo per builder and counts all public commits on that repo during the current UTC week.

---

## activity.json

This file is generated by a script.

Example:

```json
{
  "generatedAt": "2026-04-29T14:00:00Z",
  "weekStart": "2026-04-26T00:00:00Z",
  "weekEnd": "2026-05-03T00:00:00Z",
  "builders": [
    {
      "id": "alice",
      "github": "alice-dev",
      "lastActivityAt": "2026-04-29T12:34:00Z",
      "commitsThisWeek": 8,
      "reposTouchedThisWeek": 2,
      "recentCommits": [
        {
          "repo": "alice-dev/open-wallet-tool",
          "message": "Add wallet connection flow",
          "url": "https://github.com/alice-dev/open-wallet-tool/commit/abc123",
          "committedAt": "2026-04-29T12:34:00Z"
        }
      ]
    }
  ]
}
```

### Generated Activity Fields

| Field                  | Description                              |
| ---------------------- | ---------------------------------------- |
| `generatedAt`          | When the activity snapshot was created   |
| `weekStart`            | Start of the current tracking week       |
| `weekEnd`              | End of the current tracking week         |
| `lastActivityAt`       | Most recent detected GitHub activity     |
| `commitsThisWeek`      | Count of commits detected this week      |
| `reposTouchedThisWeek` | Number of repos with recent activity     |
| `recentCommits`        | Small list of recent commits for display |

## Site Pages

## 1. Dashboard Home Page

Purpose: give organizers and participants a quick overview.

Suggested sections:

### Summary Cards

- Total registered builders
- Builders active this week
- Total commits this week
- Repositories active this week
- Last dashboard update

### Builder Table

Columns:

- Builder
- Project
- GitHub
- Commits this week

## GitHub Activity Collection

The first version can use the GitHub REST API.

The collection script should:

1. Read `config/repos.yml`
2. Normalize it into `builders.json`
3. For each builder, identify the tracked repo
4. Fetch recent commits from that repo
5. Count commits in the current hackathon week
6. Write `activity.json`


---

## Automation

Use GitHub Actions to update the data automatically.

Suggested schedule:

- Run once per day
- Run manually on demand
- Commit the updated `activity.json` back to the repo
- Rebuild and deploy the static site

Example flow:

```txt
GitHub Action runs daily
→ reads builders.json
→ calls GitHub API
→ writes activity.json
→ commits updated data
→ site rebuilds
→ dashboard updates
```

Current implementation note: builder normalization now comes directly from `config/repos.yml` through a shared Node loader, while some other supporting scripts may still use shell or Ruby until later cleanup.

---

## What This Dashboard Should Not Do Yet

For the MVP, avoid:

- Complex ranking
- Automated reward calculations
- Private repo tracking
- Login systems
- Builder authentication
- Webhooks
- Real-time infrastructure
- Heavy database-backed app design

This should be an operations dashboard, not a judging machine.

---

## Important Interpretation Warning

GitHub commits are useful evidence, but they are not the full story of building.

Some meaningful hackathon work may happen outside GitHub:

- Design
- User interviews
- Documentation
- Business development
- Customer support
- Testing
- Community work
- Private planning
- No-code or low-code work

The dashboard should present GitHub activity as one signal among several.

Recommended framing:

> GitHub activity is a build signal, not a complete measure of contribution.

## MVP Success Criteria

The first version is successful if:

- Organizers can add builders by editing `builders.json`
- The site shows all builders in a readable dashboard
- GitHub activity is collected automatically
- Each builder has a profile page
- The site can be deployed as a static website
- The data files are public and inspectable
- The system reduces admin work instead of increasing it

---

## Plain-English Description

The Piece of Pie GitHub Activity Dashboard is a small static website that tracks public GitHub activity from hackathon builders. Organizers maintain a simple `builders.json` file with builder names, GitHub usernames, project links, and repo links. A scheduled GitHub Action checks those public repositories, creates an `activity.json` file, and updates the dashboard.

The dashboard gives organizers a quick way to see who is active, what repos are moving, and where each builder's work can be found. It is not a judging tool by itself. It is a lightweight operations and storytelling tool that helps make the building process more visible.
